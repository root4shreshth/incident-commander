🌐 Service Topology

frontend-bff → api-gateway ├→ order-service → payment-service │ → inventory-service │ → postgres-db ├→ user-service → auth-service └→ notification-service

📊 Per-Service Simulation

  • Health states -- healthy, degraded, unhealthy, crashed
  • Live metrics -- CPU, memory, latency p50/p99, error rate
  • Structured logs -- timestamped with realistic error patterns
  • Deployment history -- version tracking + rollback
  • Dependencies -- upstream/downstream failure cascading
  • Resource quotas -- CPU/memory limits per replica

API Reference

Base URL: https://hype4raj-incident-commander-env.hf.space

POST/resetStart a new incident episode
Request:  { "task_id": "oom_crash" }
Response:
{ "observation": { "message": "INCIDENT ALERT...", "alert": "...", "dependency_graph": {...} },
  "reward": 0.0, "done": false,
  "info": { "task_id": "oom_crash", "max_steps": 15 } }
POST/stepExecute an SRE action
Request:
{ "action_type": "read_logs", "target_service": "payment-service",
  "parameters": { "lines": 50, "severity": "ERROR" } }
Response:
{ "observation": { "message": "Logs for payment-service...", "logs": [...] },
  "reward": 0.03, "done": false, "info": { "step_count": 2 } }
GET/stateCurrent episode state
Response:
{ "episode_id": "uuid", "step_count": 5, "task_id": "oom_crash",
  "incident_resolved": true, "current_score": 1.0 }
GET/tasksList available scenarios
Response:
{ "tasks": { "oom_crash": { "difficulty": "easy", "max_steps": 15 }, ... } }
GET/healthLiveness check
Response: { "status": "ok" }

Quick Start Guide

  1. Select an incident scenario on the left (start with OOM Crash)
  2. Click "Start Incident" to begin
  3. Use list_services to see the cluster state
  4. Investigate with read_logs and check_metrics
  5. Apply a fix: restart_service, rollback_deployment, etc.
  6. Goal: maximize score (0.0 - 1.0) within the step budget
