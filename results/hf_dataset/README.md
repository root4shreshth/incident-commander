# Praetor SRE Trajectories

Production incident-response trajectories generated by the **[Praetor / IncidentCommanderEnv](https://huggingface.co/spaces/hype4raj/incident-commander-env)** RL environment. Two formats in one repo: chat-style **SFT rows** for warm-starting LLMs, and **raw step-level trajectories** with full observation + action + 6-component reward breakdown for any other use.

This dataset is the proof-of-concept for the env's data-factory thesis - production-realistic SRE trajectories at thousands per second on a laptop, instead of one per minute on a real Kubernetes cluster.

## Why this dataset exists

RL training for SRE has been gated on data, not algorithms. Real Kubernetes resets in ~60 s per episode. Microsoft's AIOpsLab requires a live K8s cluster. Even the SF OpenEnv hackathon winner trained on a real GKE cluster - heroic but not throughput-shaped. This dataset is what comes out the other end of a deterministic, seeded simulator that resets in ~0.5 ms and steps in ~0.16 ms. The 30 random-policy episodes here were generated in **0.26 seconds**.

You can use these trajectories directly, or you can regenerate them at any density by running [`scripts/export_trajectories.py`](https://github.com/root4shreshth/incident-commander/blob/main/scripts/export_trajectories.py) against the env.

## Configurations

```python
from datasets import load_dataset

# SFT chat rows - feed straight into TRL's SFTTrainer
sft = load_dataset("metamorphs/praetor-sre-trajectories", "sft", split="train")
print(sft[0]["system"][:80], "...")
print(sft[0]["assistant"][:200])

# Raw step-level rows - full obs + action + 6-component reward breakdown
raw = load_dataset("metamorphs/praetor-sre-trajectories", "raw", split="train")
print(raw[0]["task_id"], raw[0]["step"], raw[0]["reward_breakdown"])
```

## SFT format (`sft.jsonl`)

One row per (episode, step). Drop straight into `trl.SFTTrainer` with the standard chat schema.

| Field | Type | Notes |
|---|---|---|
| `system` | string | The SRE persona + action vocabulary, identical to `inference.py` |
| `user` | string | Either the alert (step 1) or the previous observation message |
| `assistant` | string | JSON action `{thinking, action_type, target_service, parameters}` |
| `task_id` | string | One of `oom_crash`, `db_pool_exhaustion`, `bad_deployment_cascade` |
| `seed` | int | Episode seed (deterministic regeneration) |
| `step` | int | 1-indexed step number within the episode |
| `source` | string | `random_policy` or `ideal_trajectory` (senior-SRE behavioral clone) |
| `episode_score` | float | Final 0-1 grader score for this episode |
| `episode_resolved` | bool | Whether the episode resolved before the step budget |

The `source = "ideal_trajectory"` rows come from `IDEAL_TRAJECTORIES` in [`incident_commander_env/server/coach.py`](https://github.com/root4shreshth/incident-commander/blob/main/incident_commander_env/server/coach.py) - these are the senior-SRE playbooks the SFT warm-start uses. Filter by `source` to use them alone.

## Raw format (`raw.jsonl`)

One row per (episode, step). Carries the full state the agent saw plus the per-step decomposed reward.

| Field | Type | Notes |
|---|---|---|
| `task_id`, `seed`, `difficulty`, `episode_id` | str/int/float/str | Episode identifiers |
| `step` | int | 1-indexed |
| `action` | object | `{action_type, target_service, parameters}` |
| `observation` | object | `{message, alert?, logs?, metrics?, services_summary?, ...}` |
| `reward_breakdown` | object | `{r_diagnostic, r_correct_op, r_resolution, r_format, r_efficiency, r_penalty}` |
| `step_message` | string | Human-readable summary of what happened |
| `episode_score` | float | Final episode score (constant across all rows in one episode) |
| `episode_resolved` | bool | Whether the episode resolved |

## Scenario families

| `task_id` | What it simulates | Hardness |
|---|---|---|
| `oom_crash` | A service crashes with OutOfMemory; agent must read logs, raise mem limit, restart | Easy |
| `db_pool_exhaustion` | Cascading 5xx from a leaking DB connection pool 2 hops upstream | Medium |
| `bad_deployment_cascade` | Bad release v2.4.0 → autoscaler → quota → starved deps; rollback before restart | Hard |

Each family is **parametric** - `(seed, difficulty)` produces a distinct instance, so the agent learns the rule, not the case.

## Generation provenance

- **Source:** Praetor v0.1.0 (`incident_commander_env`)
- **Policy:** `random_policy(rng_seed=42)` - uniform over 8 action templates
- **Families × seeds:** 3 × 10 = 30 episodes (default; configurable)
- **Difficulty:** 0.5 (mid-difficulty parametric instance)
- **Generation time:** ~0.26 s on commodity laptop
- **Reproducibility:** byte-identical with same seed; see `manifest.json` for the full config

## Ethics & limitations

- Logs and metrics are synthetic - no real customer data is present.
- The simulator covers **runtime ops** (logs, metrics, restarts, rollbacks). Code-level fixes are out of scope here; see the project's Phase-2 roadmap for `CodeAwareBackend`.
- The random policy succeeds rarely - that's intentional. Use these rows as the diversity floor, blend with `source=ideal_trajectory` for SFT warm-start.
